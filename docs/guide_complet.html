Qu'est-ce que AD Audit AI ?

Le problème qu'il résout

Dans un réseau d'entreprise, Active Directory (AD) est le cœur du système d'authentification — il gère tous les utilisateurs, ordinateurs et droits d'accès. Un attaquant qui compromet AD a un accès total à l'organisation.

Auditer manuellement un AD est long et technique : il faut analyser des milliers de relations entre comptes, groupes et machines pour trouver les "chemins d'attaque" (comment un attaquant passerait d'un compte banal à un compte Administrateur du Domaine). Une équipe d'audit peut passer plusieurs jours sur cette tâche.

AD Audit AI automatise ce travail : il ingère l'export d'un outil (BloodHound), trouve tous les chemins d'attaque, les fait analyser par une IA, les cartographie sur le référentiel MITRE ATT&CK, et génère un rapport PDF professionnel livrable directement au client.

En une phrase
AD Audit AI = BloodHound + Intelligence Artificielle + Rapport PDF — automatisé et sécurisé.

Qui utilise la plateforme ?

👤

Administrateur

Gère les utilisateurs, configure les clés LLM, accès total

📋

Manager

Crée les missions client, accède à tous les rapports

🔍

Auditeur

Upload les fichiers, consulte les chemins, télécharge les guides

Ce que la plateforme produit

  • Une liste de chemins d'attaque triés par score de risque (Critique → Faible)
  • Pour chaque chemin : explication en français, score de sévérité, techniques MITRE associées
  • Un rapport PDF multi-pages prêt à livrer au client
  • Des guides de remédiation (PowerShell) téléchargeables
  • Une visualisation graphique du réseau AD (via Cytoscape.js)

Étape 1 — Upload du fichier

L'auditeur glisse-dépose un fichier .json exporté depuis BloodHound sur la page Mission. Le frontend envoie le fichier en multipart/form-data vers POST /api/v1/engagements/{id}/analyses.

Le backend vérifie que le fichier est bien un .json et ne dépasse pas 100 MB, crée un objet Analysis en base avec le statut pending, et démarre le pipeline en arrière-plan (BackgroundTask).

Le frontend reçoit immédiatement la réponse 202 Accepted et se connecte en WebSocket pour suivre la progression.

Étape 2 — Connexion WebSocket

Le frontend ouvre une connexion WebSocket vers /ws/analyses/{id}?token=JWT. Le token est validé dès l'ouverture de la connexion (accès refusé si absent ou invalide). À chaque étape du pipeline, le backend envoie un événement JSON :

{
  "stage": "ingestion",
  "progress": 5,
  "message_fr": "Ingestion du graphe BloodHound en cours..."
}

La barre de progression dans l'interface se met à jour en temps réel.

Étape 3 — Ingestion du graphe (5% → 25%)

Le module ingestion.py lit le JSON BloodHound (formats v4 et v5 supportés). Il construit un graphe orienté avec NetworkX où :

  • Nœuds = utilisateurs, ordinateurs, groupes, domaines
  • Arêtes = relations AD (AdminTo, MemberOf, HasSession, DCSync…)

Les nœuds privilégiés sont détectés automatiquement via leur SID Windows (ex : S-1-5-21-*-512 = Domain Admins). Le graphe est aussi persisté dans Neo4j pour les requêtes Cypher avancées.

Étape 4 — Extraction des chemins d'attaque (25% → 40%)

Le module paths.py utilise l'algorithme all_simple_paths de NetworkX pour trouver tous les chemins depuis n'importe quel nœud non-privilégié vers un nœud privilégié, avec une limite de 6 sauts maximum.

Exemple de chemin : jdoe → MemberOf → IT_Support → AdminTo → PC-DEV01 → HasSession → admin_bob → MemberOf → Domain Admins

Les chemins dupliqués sont éliminés par hachage de la séquence d'arêtes.

Étape 5 — Enrichissement MITRE ATT&CK (40% → 55%)

Le module mitre.py consulte le fichier mitre_mapping.json qui associe chaque type d'arête BloodHound à une ou plusieurs techniques MITRE ATT&CK. Par exemple :

  • HasSession → T1550.002 Pass the Hash
  • DCSync → T1003.006 OS Credential Dumping
  • AdminTo → T1021.002 SMB/Windows Admin Shares

Étape 6 — Analyse LLM (55% → 85%)

Le module agent.py envoie chaque chemin au LLM (par batch de 5) avec un prompt en français détaillant le contexte du chemin. Le LLM répond avec un JSON structuré :

{
  "exploitability_score": 8.5,
  "stealth_score": 7.0,
  "global_score": 8.0,
  "risk_level": "Critique",
  "explanation": "Ce chemin permet à un attaquant...",
  "recommendation": "Supprimer le droit AdminTo sur..."
}

Si le LLM échoue 3 fois, le chemin est marqué analyse_echec (jamais de données inventées). Un score heuristique de secours est appliqué.

Étape 7 — Génération du rapport PDF (85% → 100%)

Le module report.py utilise Jinja2 pour rendre un template HTML en PDF via WeasyPrint. Le PDF inclut : page de garde, synthèse exécutive, tableau des chemins, détails par chemin critique, annexe MITRE.

Flux d'authentification

Toutes les requêtes API (sauf /auth/login, /healthz, /readyz) nécessitent un token JWT dans l'en-tête Authorization: Bearer {token}.

Connexion :
POST /api/v1/auth/login → { access_token, refresh_token }

Utilisation :
GET /api/v1/engagements
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

Renouvellement automatique (côté frontend) :
POST /api/v1/auth/refresh → nouveau access_token

Les 5 modules backend core

Ces 5 fichiers dans backend/app/modules/ sont le cœur de la plateforme. Tout le reste (API, frontend) s'appuie sur eux.

1

ingestion.py — Lecture BloodHound → Graphe

app/modules/ingestion.py

Rôle : Lire un fichier JSON exporté par BloodHound et le transformer en un graphe Python (NetworkX DiGraph) exploitable.

Ce qu'il fait concrètement :

  • Détecte si le JSON est au format BloodHound v4 ou v5 (schemas différents)
  • Crée un nœud dans le graphe pour chaque entité AD (User, Computer, Group, Domain…)
  • Crée une arête orientée pour chaque relation AD (A → AdminTo → B)
  • Marque automatiquement les nœuds privilégiés (Domain Admins = SID *-512, Enterprise Admins = *-519…)
  • Persiste le graphe dans Neo4j pour permettre des requêtes Cypher
  • Ignore les types d'arêtes inconnus sans planter (warning dans les logs)

Entrée : dict (JSON BloodHound parsé) — Sortie : (nx.DiGraph, nb_nœuds, nb_arêtes)

2

paths.py — Extraction des chemins d'attaque

app/modules/paths.py

Rôle : Trouver tous les chemins qui mènent d'un nœud non-privilégié à un nœud privilégié dans le graphe.

Algorithme : NetworkX all_simple_paths avec un cutoff de 6 sauts. Cet algorithme explore toutes les combinaisons possibles sans cycles.

Déduplication : Chaque chemin est identifié par un hachage SHA-256 de sa séquence d'arêtes. Les chemins identiques (mêmes arêtes dans le même ordre) sont ignorés.

Ce qu'un chemin contient :

AttackPathData(
  source_node = "jdoe@contoso.local",
  target_node = "DOMAIN ADMINS@contoso.local",
  hops = [{source, edge_type, target}, ...],
  length = 4,          # nombre de sauts
  source_type = "User",
  target_type = "Group",
  mitre_techniques = [...]   # rempli par mitre.py
)
3

mitre.py — Mapping MITRE ATT&CK

app/modules/mitre.py

Rôle : Enrichir chaque chemin d'attaque avec les techniques MITRE ATT&CK correspondantes.

Source : Le fichier backend/data/mitre_mapping.json qui associe chaque type d'arête BloodHound à 1-N techniques MITRE. Ce fichier est chargé au démarrage (lecture unique, résultat mis en cache).

Exemple : Un chemin avec les arêtes [MemberOf, AdminTo, HasSession] recevra :

  • MemberOf → T1069.002 (Permission Groups Discovery)
  • AdminTo → T1078.002 (Valid Accounts: Domain Accounts), T1021.002 (SMB)
  • HasSession → T1550.002 (Pass the Hash), T1558.003 (Kerberoasting)

Les doublons de techniques sont dédupliqués par chemin.

4

agent.py — Agent LLM d'analyse

app/modules/agent.py

Rôle : Envoyer chaque chemin d'attaque à un LLM (modèle de langage) pour obtenir une analyse détaillée en français.

Architecture multi-provider : Le module sélectionne le provider LLM configuré (Anthropic, OpenAI, Azure, Ollama local, Mock pour tests) via la factory dans llm_providers/. La config peut être changée en live depuis la page Paramètres sans redémarrer.

Robustesse :

  • 3 tentatives par chemin avec extraction JSON robuste (gère les blocs ```json)
  • Cache en mémoire par hachage du chemin (même chemin = jamais renvoyé au LLM)
  • En cas de quota LLM dépassé (erreur 429) : bascule immédiatement sur le score heuristique
  • Score heuristique : basé sur les types d'arêtes (DCSync/GenericAll = Critique, MemberOf seul = Faible)
  • Délai entre batches pour respecter les rate limits des providers

Prompt système : Stocké dans backend/data/prompts/analyze_path.txt. Entièrement en français.

5

report.py — Génération du rapport PDF

app/modules/report.py

Rôle : Générer un rapport PDF professionnel, livrable directement au client.

Technologie : Jinja2 (template HTML) + WeasyPrint (rendu PDF depuis HTML/CSS).

Contenu du rapport :

  1. Page de garde (nom client, code mission, date, verdict global)
  2. Sommaire automatique
  3. Synthèse exécutive (1 page, générée par un second appel LLM)
  4. Méthodologie (texte statique expliquant BloodHound + IA + MITRE)
  5. Tableau global des chemins triés par score
  6. Détail des chemins Critique et Élevé (explication + recommandation + techniques MITRE)
  7. Annexe MITRE : matrice tactique × technique
  8. Annexe conformité : mapping ISO 27001 / NIST CSF

Footer : Chaque page porte le code mission et le texte "CONFIDENTIEL — Document de travail" (configurable).

Module bonus : ldap_collector.py

Permet une collecte directe depuis un AD live via LDAP (sans passer par BloodHound). Activé uniquement avec le flag --allow-live. Trace complète dans les audit_logs. Utile pour les missions où on a accès direct au réseau cible.

Module bonus : remediation.py

Génère des guides de remédiation en Markdown avec commandes PowerShell validées, explications en français pour chaque type d'arête dangereuse. Disponible à l'unité (par chemin) ou en bundle ZIP (toute l'analyse).

Le frontend page par page

Le frontend est en HTML/CSS/JavaScript pur, sans framework ni étape de build. Alpine.js (chargé depuis CDN) gère la réactivité, Chart.js les graphiques. Tout est servi par Nginx depuis le conteneur frontend.

Page 1 — Login (/index.html)

Rôle : Authentification de l'utilisateur.

Comportement :

  • Formulaire email + mot de passe
  • Appel POST /api/v1/auth/login
  • Stockage du JWT dans localStorage
  • Redirection automatique vers le dashboard si déjà connecté
  • Message d'erreur en français si identifiants incorrects
// js/auth.js — gère la session
auth.login(email, password)
  → api.post('/auth/login', ...)
  → localStorage.setItem('token', ...)
  → window.location = '/dashboard.html'

Page 2 — Dashboard (/dashboard.html)

Rôle : Vue d'ensemble de toutes les missions d'audit.

Ce que la page affiche :

  • Statistiques globales : nombre de missions par statut (En cours, Terminé, Archivé)
  • Nombre total de chemins d'attaque et de chemins critiques trouvés sur toutes les missions
  • Kanban 3 colonnes : À faire / En cours / Terminé — les cartes de mission peuvent être glissées entre colonnes
  • Bouton "Nouvelle mission" pour créer un engagement

Données chargées via : GET /api/v1/engagements + GET /api/v1/engagements/stats/summary

Page 3 — Mission (/engagement.html?id=UUID)

Rôle : Gérer une mission spécifique et ses analyses.

Ce que la page affiche :

  • Informations de la mission (client, code, statut, dates)
  • Zone de notes libres (éditeur Markdown simple)
  • Zone de drag & drop pour uploader un fichier BloodHound .json
  • Liste des analyses avec statut et progression
  • Barre de progression temps réel (WebSocket) pendant l'analyse
  • Boutons : Voir les chemins, Télécharger le rapport PDF, Télécharger les guides ZIP
  • Visualisation graphique du réseau AD (Cytoscape.js) une fois l'analyse terminée

Page 4 — Détail d'un chemin (/path_detail.html?id=UUID)

Rôle : Afficher l'analyse complète d'un chemin d'attaque spécifique.

Ce que la page affiche :

  • Le chemin visuellement : nœud source → arêtes → nœud cible
  • Scores : Exploitabilité, Furtivité, Score global (jauges visuelles)
  • Critique badge de niveau de risque
  • Explication en français de l'analyste IA (pourquoi ce chemin est dangereux)
  • Recommandation concrète de remédiation en français
  • Badges des techniques MITRE ATT&CK (cliquables → lien vers attack.mitre.org)
  • Bouton pour télécharger le guide PowerShell de remédiation

Page 5 — Paramètres (/settings.html)

Rôle : Configuration du provider LLM (accessible uniquement aux admins).

Fonctionnalités :

  • Sélection du provider : Anthropic Claude, OpenAI GPT, Azure OpenAI, Ollama (local), etc.
  • Saisie et validation des clés API
  • Test de connexion en temps réel
  • Sélection du modèle (ex: claude-sonnet-4-5)

Concepts clés

Active Directory (AD)

Active Directory est le service d'annuaire de Microsoft, présent dans la quasi-totalité des entreprises. Il centralise la gestion de :

  • Utilisateurs (comptes humains et services)
  • Ordinateurs (postes de travail, serveurs)
  • Groupes (collections d'utilisateurs ou d'ordinateurs)
  • Droits d'accès (qui peut faire quoi sur quoi)

Le compte le plus puissant est le Domain Admin — qui a le contrôle total de tout le domaine Windows. La mission d'un auditeur AD est de trouver comment un attaquant pourrait l'atteindre.

BloodHound

BloodHound est un outil open-source qui collecte les relations AD (via LDAP) et les visualise sous forme de graphe. Il exporte ces données en JSON.

AD Audit AI consomme cet export JSON comme point d'entrée. Aucun accès direct à l'AD de production n'est nécessaire — l'auditeur fournit l'export et la plateforme fait le reste.

Analogie simple
BloodHound = carte du réseau. AD Audit AI = GPS qui trouve tous les chemins dangereux sur cette carte, en explique les risques, et génère le rapport de navigation.

Chemins d'attaque

Un chemin d'attaque est une séquence de relations AD qu'un attaquant pourrait enchaîner pour escalader ses privilèges. Par exemple :

jdoe (Utilisateur standard)
  → MemberOf → IT_Support (Groupe)
  → AdminTo → PC-DEV-01 (Ordinateur)
  → HasSession → admin_svc (Compte Service)
  → MemberOf → Domain Admins (Groupe privilégié)

Chaque flèche est une arête BloodHound, chaque ellipse est un nœud. Ce chemin fait 4 sauts et permet d'atteindre les Domain Admins depuis un compte ordinaire.

MITRE ATT&CK

MITRE ATT&CK est un référentiel mondial des tactiques et techniques utilisées par les attaquants. Chaque technique a un identifiant (ex: T1078.002) et est documentée avec des détections et mitigations.

Tactiques principales en AD :

TactiqueDescriptionExemple technique
DiscoveryDécouverte du réseauT1087.002 Account Discovery
Lateral MovementDéplacement latéralT1021.002 SMB Admin Shares
Credential AccessVol de credentialsT1003.006 DCSync
Privilege EscalationÉlévation de privilègesT1078.002 Valid Accounts
PersistencePersistanceT1098 Account Manipulation

LLM (Large Language Model)

Un LLM est un modèle d'IA capable de comprendre et générer du texte. AD Audit AI utilise un LLM (par défaut Claude d'Anthropic) pour analyser chaque chemin d'attaque et produire :

  • Une explication en français adaptée au niveau du client (pas de jargon technique brut)
  • Un score de risque basé sur l'exploitabilité et la furtivité
  • Une recommandation concrète de remédiation

Le mode Ollama permet de faire tourner un LLM entièrement en local (Llama 3, Mistral) — aucune donnée ne quitte l'infrastructure.

JWT (JSON Web Token)

Le JWT est le mécanisme d'authentification utilisé par l'API. C'est un token signé qui contient l'identité de l'utilisateur et son rôle. Il est valide 15 minutes (access token) et se renouvelle automatiquement via un refresh token de 7 jours.

Structure d'un JWT :
Header.Payload.Signature

Payload décodé :
{
  "sub": "uuid-de-l-utilisateur",
  "role": "auditor",
  "exp": 1748000000  // timestamp d'expiration
}

Guide d'utilisation pas à pas

Prérequis

  • Docker Desktop installé et démarré
  • Un fichier .env créé à partir de .env.example (au minimum : ANTHROPIC_API_KEY)
  • Le dossier ad-audit-ai/ cloné

Étape 1 — Démarrer la stack

# Copier le fichier de configuration
cp .env.example .env
# Éditer .env : mettre votre clé Anthropic
nano .env  # ou notepad .env sur Windows

# Démarrer tous les services
docker compose up -d

# Vérifier que tout est sain (attendre 30-60s)
docker compose ps
Premier démarrage
PostgreSQL, Neo4j et le backend démarrent. Le backend crée automatiquement les tables et l'utilisateur admin défini dans .env (par défaut : admin@adauditai.local / ChangeMeNow!2026).

Étape 2 — Se connecter

Ouvrir https://localhost (ou http://localhost en dev). Ignorer l'avertissement de certificat auto-signé. Se connecter avec les identifiants admin.

Étape 3 — Créer un utilisateur auditeur (optionnel)

En tant qu'admin, via la page Paramètres ou directement via l'API :

POST /api/v1/auth/register
Authorization: Bearer {token_admin}
{
  "email": "auditeur@entreprise.fr",
  "password": "MotDePasseForte123!",
  "full_name": "Jean Dupont",
  "role": "auditor"
}

Étape 4 — Créer une mission

Sur le dashboard, cliquer "Nouvelle mission" et remplir :

  • Nom du client : ex. "Acme Corp"
  • Code mission : ex. "AUDIT-2026-0042" (unique)
  • Description : contexte de la mission

Étape 5 — Uploader le fichier BloodHound

Sur la page de la mission, glisser-déposer le fichier .json exporté depuis BloodHound (ou utiliser le bouton "Parcourir"). Le pipeline démarre automatiquement.

Pour tester : utiliser le fichier exemple fourni : backend/data/sample_graph.json

Étape 6 — Suivre la progression

La barre de progression se met à jour en temps réel via WebSocket :

  • 5% — Ingestion du graphe BloodHound
  • 25% — Extraction des chemins d'attaque
  • 40% — Enrichissement MITRE ATT&CK
  • 55% — Analyse par l'agent IA
  • 85% — Sauvegarde des résultats
  • 100% — Terminé !
Durée
En mode mock LLM (tests) : ~10 secondes. Avec un vrai LLM sur un graphe de 100 nœuds : 2-5 minutes selon le provider et le nombre de chemins.

Étape 7 — Explorer les résultats

Une fois l'analyse terminée :

  • Cliquer "Voir les chemins" → tableau paginé avec filtres (par niveau de risque, score, technique MITRE)
  • Cliquer sur un chemin → page de détail complète
  • Explorer la visualisation graphique (Cytoscape.js) sur la page Mission

Étape 8 — Télécharger le rapport

Bouton "Télécharger le rapport PDF" sur la page Mission → fichier PDF multi-pages généré à la volée.

Étape 9 — Guides de remédiation

  • Par chemin : bouton "Guide de remédiation" sur la page Détail → fichier Markdown avec commandes PowerShell
  • Tout d'un coup : bouton "Bundle remédiation ZIP" sur la page Mission → archive avec tous les guides

Changer le provider LLM

Page Paramètres (admin uniquement) → sélectionner le provider → entrer la clé API → Tester la connexion → Sauvegarder. Le changement est pris en compte immédiatement, sans redémarrage.

Suggestions d'amélioration — V2

Voici 20 idées concrètes, classées par impact et effort, pour faire passer la plateforme au niveau supérieur :

Fonctionnalités haute valeur

📊

Mode diff — Comparaison avant/après remédiation

Permettre de comparer deux analyses d'un même client (avant et après corrections). Afficher quels chemins ont été supprimés, lesquels persistent, et le gain de sécurité en pourcentage. Extrêmement précieux pour démontrer la valeur de la mission au client.

Impact : Élevé
📈

Score de maturité AD (0-100)

Calculer un score global de sécurité AD basé sur le nombre et la sévérité des chemins. Suivre son évolution dans le temps sur un graphique de tendance. Remplacer le tableau de chemins par une vue "scorecard" pour les présentations exécutives.

Impact : Élevé
🎯

Tier Model Microsoft (T0/T1/T2)

Classifier automatiquement les nœuds selon le modèle de niveaux de Microsoft (T0 = DC et comptes admin critiques, T1 = serveurs applicatifs, T2 = postes utilisateurs). Mettre en évidence les violations de tier (ex : T2 → T0 en 3 sauts).

Impact : Élevé
🔗

Connecteur Azure AD / Entra ID

Étendre la collecte aux environnements cloud Microsoft (Azure AD, Entra ID). De plus en plus d'entreprises sont en hybride on-prem + cloud. Ajouter les relations Entra-spécifiques (AppRoleAssignment, ConditionalAccess bypass, etc.).

Impact : Élevé
📤

Export STIX 2.1 pour SIEM

Exporter les chemins d'attaque et techniques MITRE au format STIX 2.1 (standard CTI). Permet l'intégration avec des SIEM comme Splunk, Microsoft Sentinel ou Elastic SIEM pour corréler les chemins détectés avec les alertes en cours.

Impact : Moyen
👥

Mode collaboratif multi-auditeur

Permettre à plusieurs auditeurs d'annoter les chemins (commentaires, statuts "en cours de remédiation", "faux positif", "accepté"). Notifications en temps réel sur les mises à jour de collègues. Parfait pour les missions en équipe.

Impact : Moyen

Amélioration de l'analyse IA

🧠

Fine-tuning sur données validées

Collecter les annotations des auditeurs (correction des scores, reformulation des explications) pour fine-tuner un modèle local (Llama 3, Mistral) spécialisé AD. Réduire les coûts API tout en améliorant la précision des analyses.

Impact : Élevé long terme
⚡

Analyse parallèle des chemins

Traiter les batches LLM en parallèle avec asyncio.gather() au lieu de séquentiellement. Réduire le temps d'analyse de 10x sur un graphe de 100 chemins (de ~10 min à ~1 min avec un provider permissif).

Impact : Performance
🔄

Re-analyse ciblée

Permettre de re-lancer l'analyse LLM sur un seul chemin (si le résultat initial semble incorrect) sans refaire toute l'analyse. Ajouter un bouton "Ré-analyser" sur la page de détail.

Impact : UX

UX et présentation

🌙

Mode sombre

Ajouter un thème sombre (dark mode) pour le travail de nuit ou les préférences personnelles. Utiliser CSS custom properties déjà en place — il suffit d'overrider les variables dans une classe .dark.

Impact : UX
📱

Application mobile (PWA)

Transformer le frontend en Progressive Web App (PWA) pour consultation sur téléphone lors des réunions client. Notifications push quand une analyse se termine.

Impact : Mobilité
🎨

Rapport personnalisable

Permettre à l'auditeur de sélectionner les sections à inclure dans le PDF (ex : "inclure annexe conformité ? oui/non"), choisir le logo client, modifier le texte de couverture. Rendre le rapport 100% sur-mesure.

Impact : Commercial

Sécurité et opérations

🔐

Chiffrement des clés API en base

Les clés API LLM sont actuellement stockées en clair dans PostgreSQL. Ajouter un chiffrement Fernet (AES-256) avec une clé dérivée de APP_SECRET_KEY. En cas de fuite de la base, les clés restent inutilisables.

Impact : Sécurité critique
🚦

Rate limiting sur l'API

Ajouter slowapi (basé sur redis) pour limiter le nombre de requêtes par IP/utilisateur. Protéger l'endpoint de login contre le brute-force (max 10 tentatives / 15 min par IP).

Impact : Sécurité
🔔

Notifications Slack / Email

Envoyer une notification quand une analyse se termine ou échoue. Alerter si un chemin Critique est découvert. Utile pour les analyses longues lancées la nuit.

Impact : Productivité
📥

Visualiseur d'audit logs

Ajouter une page admin pour consulter les audit_logs avec filtres (par utilisateur, par action, par période). Export CSV pour les équipes conformité. Actuellement les logs sont en base mais pas exposés dans l'UI.

Impact : Conformité
🌐

Multi-langue

Ajouter le support de l'anglais pour les équipes internationales (les rapports seraient en EN ou FR selon la config de la mission). Utiliser i18n côté frontend et paramètrer la langue du prompt LLM par mission.

Impact : International
🤝

Intégration ticketing (Jira, ServiceNow)

Créer automatiquement des tickets dans Jira ou ServiceNow pour chaque chemin critique, avec le score, l'explication, et la recommandation. Permet de suivre la remédiation dans les outils existants du client.

Impact : Workflow entreprise
📋

Templates de missions

Créer des templates de mission prédéfinis (ex : "Audit initial", "Contrôle post-remédiation", "Audit annuel") avec les paramètres LLM, les filtres de chemins et les sections de rapport préconfigurés. Réduire le temps de mise en place.

Impact : Productivité
